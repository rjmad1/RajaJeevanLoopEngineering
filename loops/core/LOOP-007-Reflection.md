**Loop ID:** LOOP-007  
**Name:** Reflection  
**Version:** 1.0  
**Status:** Active  
**Category:** Core  
**Depends On:** LOOP-001 — Architecture Discovery, LOOP-002 — Context Assembly, LOOP-003 — Task Discovery, LOOP-004 — Planning, LOOP-005 — Implementation, LOOP-006 — Verification

## Purpose

LOOP-007 captures the engineering knowledge produced by each completed engineering cycle. Its responsibility is organisational learning: analysing what was planned, what was implemented, what was verified, and what the differences between these reveal about the quality of the engineering process. It does not modify implementation artefacts. It does not re-verify the implementation. It feeds what should be remembered into the **Knowledge Graph** so that future runs of every loop in the chain are better than the current one.

The loop is the mechanism by which the engineering operating system improves over time. Without it, each cycle starts from the same baseline as the last. With it, patterns accumulate, recurring failures become visible before they recur, and the knowledge embedded in each completed task propagates forward to the Knowledge Graph. Upon updating the Graph, LOOP-007 triggers the **Knowledge Integrity Steward (LOOP-504)** to automatically synchronize downstream Markdown files, ADRs, and the roadmap.

---

## Problem Statement

Engineering knowledge generated by a completed task is the most perishable asset in software development. A lesson learned at 3 p.m. on Thursday is lost by the next sprint if it is not captured as durable, reusable knowledge. Planning gaps that caused a LOOP-004 replanning event repeat in the next task. Verification failures that revealed a missing test category are not reflected in the next LOOP-006 run. Implementation patterns that worked exceptionally well are not carried forward. The cost of not reflecting is paid silently, in cycles of preventable rework, repeated architectural mistakes, and accumulated technical debt that was observable but never recorded.

---

## Why This Loop Exists

Reflection is the learning layer of the engineering operating system. Each of the preceding loops is an execution loop: it performs a defined operation and produces defined outputs. LOOP-007 is an analysis loop: it consumes the outputs of all preceding loops as evidence and produces knowledge. The distinction matters because knowledge has a different lifecycle than execution artefacts — it accumulates, refines, and supersedes, rather than being produced and consumed once. A separate loop with its own knowledge base, update protocol, and quality gates is the minimum structure that makes knowledge accumulation durable rather than incidental.

---

## Scope

**In scope:**
- Analysing the completed engineering cycle across all loops (LOOP-001 through LOOP-006) for a selected task
- Comparing planned outcomes (LOOP-004) with actual outcomes (LOOP-005) and verified outcomes (LOOP-006)
- Recording lessons learned with evidence citations
- Capturing reusable engineering patterns identified during the cycle
- Identifying process improvements for future loop runs
- Documenting newly discovered risks not in the existing risk registers
- Recording architectural observations that should influence future LOOP-001 or LOOP-004 runs
- Updating `docs/loops/core/SKILL-*.md` files where specific skill observations were made
- Updating `docs/loops/core/STATUS-007.md`
- Producing the complete reflection knowledge base in `docs/reflection/`

**Out of scope:**
- Modifying any repository source file, test file, configuration file, or planning artefact — this constraint is absolute
- Re-verifying the implementation (LOOP-006's domain)
- Prescribing specific implementation solutions for improvement opportunities (LOOP-003 and LOOP-004 act on the opportunities identified here)
- Making architectural decisions (observations are recorded; decisions are made in LOOP-001 and LOOP-004)
- Deploying or releasing the completed implementation
- Updating `docs/architecture/` artefacts directly — architectural observations are recorded in `docs/reflection/`; LOOP-001 decides whether and how to update the architecture knowledge base
- Removing or superseding entries in other loops' SKILL files without those loops' own update mechanisms; this loop records recommendations, not unilateral changes

**Maximum run duration:** 1 hour. If the loop has not reached a Stop Condition within this window, it halts with `status: stopped`, produces a partial Reflection, and records which analysis categories were not reached.

### Versioning of Accumulative Documents

`lessons-learned.md`, `engineering-patterns.md`, and `improvement-opportunities.md` are accumulative documents: each LOOP-007 run appends to them rather than replacing them. Each entry carries: a run ID, task ID, task type, date, and entry ID (`LESSON-NNN`, `PAT-NNN`, `IMP-NNN`). Entries from prior runs are never deleted; they may be superseded (a newer entry explicitly marks an older one obsolete by ID reference) or confirmed (a newer entry cites an older one as validated by a second observed cycle).

`decision-log.md` and `reflection-summary.md` are replaced on each run — they reflect the current cycle only.

## Trigger

A run is initiated by any of the following:

1. **LOOP-006 completion** — LOOP-006 has produced a final outcome (any of the three valid outcomes) and the reflection loop is next in the engineering chain.
2. **Manual invocation** — An engineer explicitly triggers reflection for a completed task.
3. **Rework cycle completion** — A task has completed a full rework cycle (LOOP-006 Requires Rework → LOOP-005 rework → LOOP-006 re-verification → Accepted), providing comparative evidence across multiple LOOP-005/006 iterations.

Trigger source, task ID, LOOP-006 outcome, and timestamp must be recorded in `STATUS-007.md` at run start.

## Reflection Principles

### Principle 1 — Evidence-Based Reflection

Every lesson, pattern, and improvement proposal must be supported by specific evidence from the completed cycle's artefacts. Evidence means: a finding ID, a step ID, a metric value, a deviation record, a gate outcome, or a direct quotation from a planning or implementation document. An observation without evidence is not a lesson — it is an opinion. Opinions are not recorded in the knowledge base.

### Principle 2 — Blameless Analysis

Reflection analyses process failures, not individual failures. The question is never "who caused this" but "what in the process allowed this to occur and what would prevent its recurrence." Every lesson is phrased as a process or structural observation, not a performance assessment. The blameless frame is not diplomatic softening; it is analytical precision — process causes are actionable; personal causes are not.

### Principle 3 — Reusable Knowledge Capture

A lesson that applies only to this specific task at this specific moment has no knowledge value. The test for every lesson is: "Will knowing this make the next task better?" If yes, record it with its applicability scope (task category, component type, repository size, technology class). If no, it is an anecdote, not a lesson. Anecdotes are not recorded in the knowledge base; they belong in the run's Reflection document only.

### Principle 4 — Continuous Improvement

Improvement proposals generated by this loop become the inputs to future LOOP-003 and LOOP-004 runs. They are not aspirational notes. Every improvement proposal must have a target loop (the loop whose behaviour it would change), a specific mechanism (what the target loop would do differently), and a priority (derived from the severity of the failure or gain it addresses).

### Principle 5 — Repository-Wide Learning

Knowledge captured here is scoped to be applicable beyond the current task. Where a lesson applies to a specific module, that scope is declared. Where a lesson applies to all tasks of a specific category, that scope is declared. Where a lesson applies to all tasks in this repository, that is the declared scope. The scope determines how the lesson propagates into future LOOP-002 context assemblies and LOOP-004 planning sessions.

### Principle 6 — Traceability

Every entry in the knowledge base carries: the cycle run ID, the task ID, the LOOP-006 outcome, and the specific artefact reference (finding ID, step ID, or metric name) that produced it. This traceability enables future audits to determine whether a lesson is still valid, whether it has been superseded, and what evidence it was originally based on.

### Principle 7 — Non-Duplication

Before adding a new lesson or pattern, the prior `lessons-learned.md` and `engineering-patterns.md` are checked for an existing entry covering the same observation. If an exact match exists, the existing entry is confirmed (a `confirmed_by: [run-id]` annotation is added). If a partial match exists, the new entry either supersedes the prior one or is recorded as a refinement (narrower or broader scope). The knowledge base grows in precision, not in volume.

### Principle 8 — Long-Term Maintainability

Each accumulative document (`lessons-learned.md`, `engineering-patterns.md`, `improvement-opportunities.md`) is designed to be readable and actionable after many cycles. Entries that have been superseded are marked `obsolete` with a reference to the superseding entry. Entries that have not been observed in more than ten cycles are marked `unconfirmed`. Entries marked `obsolete` or `unconfirmed` are not removed — they are historical records — but they are not surfaced as active guidance in future LOOP-002 context selections.

## Reflection Categories

The following ten categories are analysed in every run. For each category, the loop produces a category assessment: a summary of observations, the evidence supporting each observation, and the lessons and patterns identified.

### Category A — Planning Effectiveness

**Purpose:** Evaluate the quality of the LOOP-004 plan against what was actually required to implement the task.  
**Evidence sources:** `execution-plan.md`, `implementation-summary.md` (planned-vs-actual section), `assumptions.md`, `STATUS-005.md`.  
**Analysis questions:**
- How many plan steps were executed as declared versus how many required deviation?
- How many planning assumptions from LOOP-004 were refuted during LOOP-005?
- Were planning assumptions refuted because of missing context (a LOOP-002 gap) or because of unforeseen complexity (a LOOP-004 estimation gap)?
- Were the atomicity standards (LOOP-004 criteria) appropriate for the task's actual complexity?
- Were the parallelisation group assignments correct in practice?
- Did the dependency ordering rules produce the correct execution sequence, or did hidden dependencies emerge?
- How many plan steps required replanning?

**Lesson type produced:** `planning-gap` or `planning-success`.

### Category B — Implementation Effectiveness

**Purpose:** Evaluate the quality of LOOP-005 execution against the plan.  
**Evidence sources:** `change-log.md`, `implementation-metadata.md`, `assumptions.md`, `STATUS-005.md`.  
**Analysis questions:**
- What was the ratio of completed to deferred plan steps?
- How many new assumptions were recorded during implementation beyond those in the planning register?
- Were deviations from the plan caused by plan incompleteness or by implementation complexity?
- Were documentation obligations fulfilled or deferred?
- Were there unauthorised modification attempts blocked?
- What was the blast radius of the implementation compared to the planned scope?
- Did the implementation produce any patterns (successful approaches, tool sequences, code structures) that are reusable for similar tasks?

**Lesson type produced:** `implementation-gap`, `implementation-success`, or `implementation-pattern`.

### Category C — Verification Effectiveness

**Purpose:** Evaluate the quality and coverage of the LOOP-006 verification.  
**Evidence sources:** `verification-report.md`, `quality-gates.md`, `verification-metadata.md`, `regression-report.md`.  
**Analysis questions:**
- What was the verification confidence score and what drove its value?
- Which finding severity levels appeared, and which verification categories produced them?
- Were any findings false positives (implementation was correct; criterion or method was wrong)?
- Were non-deterministic results observed? Were they explained?
- Did regression detection find regressions, and were they attributed correctly?
- Which quality gate criteria were most discriminating (most frequently produced findings)?
- Were there verification categories that were consistently not-applicable for this task type, suggesting they could be conditionally skipped in future runs?

**Lesson type produced:** `verification-gap`, `verification-success`, or `verification-calibration`.

### Category D — Architecture Observations

**Purpose:** Record observations about the architectural state of the repository revealed during this cycle.  
**Evidence sources:** `architecture-validation.md`, `docs/architecture/module-catalog.md`, `docs/architecture/dependency-map.md`, ADR directory.  
**Analysis questions:**
- Were any module boundaries observed to be misaligned with the catalog?
- Were any undeclared dependencies discovered?
- Were any ADRs found to conflict with current implementation patterns?
- Were any architectural patterns observed (positive: a clean abstraction; negative: a coupling that made implementation harder than expected)?
- Did the architecture knowledge base (LOOP-001 outputs) accurately support the needs of this task?

**Lesson type produced:** `architecture-observation` or `architecture-risk`.

### Category E — Documentation Observations

**Purpose:** Record observations about documentation quality and completeness.  
**Evidence sources:** Category 5 results in `quality-gates.md`, `change-log.md` (documentation steps), `implementation-summary.md` (outstanding limitations).  
**Analysis questions:**
- Were documentation obligations fulfilled proactively (within the same step as the code change) or deferred?
- How many documentation gaps were detected by LOOP-006?
- Are there patterns in which types of documentation gaps recur (API catalog, event catalog, README, inline docs)?
- Was the documentation quality sufficient for LOOP-006 to perform its verification?

**Lesson type produced:** `documentation-gap`, `documentation-pattern`, or `documentation-success`.

### Category F — Technical Debt

**Purpose:** Record technical debt observations discovered or confirmed during this cycle.  
**Evidence sources:** Implementation assumptions from `assumptions.md`, Informational findings in `verification-report.md`, deviation records from `STATUS-005.md`, architectural observations from `architecture-validation.md`.  
**Analysis questions:**
- What technical debt was observed but not addressed (intentionally, because it was out of scope)?
- Was any technical debt addressed as part of this task that was not the primary objective?
- Did existing technical debt make this task harder than it would otherwise have been (higher step count, more assumptions, more deviations)?
- Is the observed debt structural (affects module boundaries or architecture) or local (affects single files or functions)?

**Lesson type produced:** `debt-observation`. Debt observations do not prescribe remediation — they become candidate LOOP-003 task inputs.

### Category G — Engineering Process Improvements

**Purpose:** Identify specific, actionable improvements to the loop engineering process.  
**Evidence sources:** Gate outcomes across all loops, deviations and blockers in STATUS files, rework cycle count, confidence score drivers.  
**Analysis questions:**
- Which loop in the chain produced the most GATE-1 triggers? Is this expected for the task category or does it signal a process gap?
- Were any Emergency Stops triggered? Were they preventable through earlier detection?
- Which part of the process produced the longest elapsed time? Is the duration appropriate or a symptom of inefficiency?
- Were there steps in any loop that were consistently not-applicable — candidates for conditional execution?
- Did the Maker/Checker pattern function as intended, or were there near-misses where the same agent would have verified its own work?

**Lesson type produced:** `process-improvement` (target loop specified).

### Category H — Automation Opportunities

**Purpose:** Identify engineering activities performed manually in this cycle that could be supported by tooling or automation.  
**Evidence sources:** Steps in any loop that required manual file inspection, manual cross-referencing, or manual count verification.  
**Analysis questions:**
- Were there verification checks executed manually that could be implemented as deterministic tests?
- Were there documentation consistency checks that could be automated?
- Were there architecture validation checks (module boundary, dependency rule) that could be expressed as static analysis rules?
- Were there assumption-recording tasks that could be partially pre-populated from the plan?

**Lesson type produced:** `automation-opportunity`. Automation opportunities are the highest-value lesson type because they permanently reduce the human and agent effort per cycle.

### Category I — Knowledge Gaps

**Purpose:** Identify gaps in the engineering knowledge base that made this cycle harder than necessary.  
**Evidence sources:** LOOP-001 unknowns, LOOP-002 context gaps, planning assumptions that were refuted, verification categories that lacked evidence.  
**Analysis questions:**
- What did the cycle assume was known that turned out to be unknown or wrong?
- What architectural knowledge would have prevented the top planning deviation?
- What context would have prevented the top implementation assumption?
- Are there recurring knowledge gaps across tasks of the same category?

**Lesson type produced:** `knowledge-gap`. Knowledge gaps feed directly back to LOOP-001 (architecture discovery priorities) and LOOP-002 (context assembly selection rules).

### Category J — Risk Observations

**Purpose:** Record newly observed risks not in the existing risk registers.  
**Evidence sources:** GATE-1 triggers, Critical and High findings, deviation types, assumption refutations.  
**Analysis questions:**
- Did any risk materialise that was not in any loop's risk register?
- Were the existing risk mitigations (gate controls, verification categories) sufficient to contain the risks that materialised?
- Are there risks that were in the register but never triggered — potentially over-estimated?

**Lesson type produced:** `risk-observation`. New risks are added as candidates to the appropriate loop's risk register via LOOP-004 in the next planning cycle.

## Workflow

### Step 1 — Load Completed Execution Artefacts

**Agent:** `REFLECT-AGENT`  
**Inputs:** All required input files  
**Outputs:** In-memory evidence index  

Load all required artefacts. Build a cross-referenced evidence index: for each loop, load its STATUS file, SKILL file, and primary output artefacts. Index entries by loop, step ID, finding ID, gate outcome, and metric.

Check for prior reflection artefacts (`lessons-learned.md`, `engineering-patterns.md`, `improvement-opportunities.md`). If they exist, load them and build a prior-entry index for deduplication in Step 6.

Assess evidence completeness: for each required input, record whether it is present, partial, or absent. Partial or absent inputs are recorded as evidence gaps and narrow the reflection scope accordingly. They are also the first candidates for lessons (a missing artefact is itself a process observation).

Record the LOOP-006 outcome (`accepted`, `requires_rework`, `requires_human_review`, or a stopped run) from `STATUS-006.md`. The outcome frames the entire reflection: a Requires Rework outcome directs more attention to Categories B and C; an Accepted outcome with a low confidence score directs attention to Category C.

---

### Step 2 — Compare Planned Versus Actual Outcomes

**Agent:** `REFLECT-AGENT`  
**Inputs:** `execution-plan.md`, `implementation-summary.md` (planned-vs-actual section), `verification-summary.md`  
**Outputs:** Planned-versus-actual comparison record  

Construct the planned-versus-actual comparison across three dimensions:

**Scope dimension:** What was planned to be changed versus what was actually changed. Source: the execution plan's `files_modified` lists versus `change-log.md`. Metrics: count of planned files, count of actual files, count of files in plan but not changed (deferred), count of files changed but not in plan (deviations blocked or approved).

**Assumption dimension:** What the plan assumed to be true versus what was found to be true during implementation. Source: LOOP-004 planning assumptions (`ASSM-NNN`) versus LOOP-005 assumption resolution records. Metrics: count validated, count refuted, count unresolved.

**Quality dimension:** What the plan expected the verification to find versus what it actually found. Source: the verification plan's declared checkpoints versus LOOP-006's quality gate results. Metrics: count passed, count failed, count not-reached, confidence score.

The comparison record is the primary evidence input for Categories A, B, and C. Every observation in those categories must be traceable to a specific data point in this comparison.

---

### Step 3 — Identify Successful Patterns

**Agent:** `REFLECT-AGENT`  
**Inputs:** Planned-versus-actual comparison, all category evidence sources  
**Outputs:** Candidate successful pattern list  

A successful pattern is an approach — to planning, implementation, or verification — that produced a measurably better outcome than the baseline. Evidence of success includes: zero deviations in a step type, zero findings in a verification category, a verified assumption, a gate that was triggered and resolved without rework, or a clean regression scope with no attributed regressions.

For each identified successful pattern, record:

| Field | Content |
|-------|---------|
| `pattern_id` | `PAT-NNN` |
| `pattern_type` | `planning`, `implementation`, `verification`, `architecture`, `documentation`, `process` |
| `title` | A concise, reusable title (e.g., "Interface-before-implementation ordering prevents hidden dependency regressions") |
| `context` | The conditions under which this pattern applies: task category, module type, technology class, or other scoping |
| `evidence` | The specific artefact references that confirm the pattern produced a positive outcome |
| `applicability` | `task-type: [list]` or `all-tasks` or `component-type: [list]` |
| `confidence` | `confirmed` (observed in ≥2 cycles), `likely` (observed in this cycle, consistent with prior experience), `candidate` (observed once, insufficient to generalise) |
| `prior_entry_ref` | `PAT-NNN` of the prior entry this confirms or refines (if any) |

Successful patterns with `confidence: candidate` are recorded but not surfaced as active guidance until confirmed in a second cycle.

---

### Step 4 — Identify Unsuccessful Patterns

**Agent:** `REFLECT-AGENT`  
**Inputs:** Planned-versus-actual comparison, findings register, deviation records, gate outcomes  
**Outputs:** Candidate unsuccessful pattern list (anti-patterns)  

An unsuccessful pattern is an approach that produced a measurably worse outcome than expected. Evidence of failure includes: a GATE-1 trigger from a plan deviation, a `Critical` or `High` finding in the verification report, an attributed regression, a refuted planning assumption, a rework cycle, or a stopped loop.

For each identified unsuccessful pattern, record the same fields as a successful pattern, plus:

| Field | Content |
|-------|---------|
| `failure_mode` | The specific mechanism by which the pattern produced a negative outcome |
| `detection_point` | The step or gate at which the failure became visible |
| `cost` | The estimated impact: rework cycle count, gate delays, confidence score reduction, or number of findings generated |
| `prevention_mechanism` | The specific process change, planning rule, or verification check that would prevent this pattern in future cycles |

The prevention mechanism must be specific: not "plan more carefully" but "add a precondition-check step for shared interface changes when the affected module has more than two declared consumers." The prevention mechanism becomes the content of the improvement proposal generated in Step 7.

---

### Step 5 — Record Lessons Learned

**Agent:** `REFLECT-AGENT`  
**Inputs:** Category A through J assessments, successful and unsuccessful pattern lists  
**Outputs:** Candidate lesson list  

For each category, derive lessons from the pattern analysis. A lesson is a concise, actionable statement that future agents and engineers can apply. It has a specific structure:

| Field | Content |
|-------|---------|
| `lesson_id` | `LESSON-NNN` |
| `category` | Category letter (A through J) |
| `title` | A statement in active voice: what to do or what to avoid |
| `context` | When this lesson applies (task category, module type, conditions) |
| `evidence` | Specific artefact references from this cycle that support the lesson |
| `mechanism` | Why this lesson is true: the causal chain from the condition to the outcome |
| `action` | The specific change in future behaviour: in which loop, at which step, doing what differently |
| `confidence` | `confirmed`, `likely`, or `candidate` (same scale as patterns) |
| `prior_entry_ref` | `LESSON-NNN` of a prior entry this confirms or supersedes (if any) |

**Lesson quality checks (applied before recording):**

1. Does the lesson have a cited evidence reference? If no: it is an opinion; do not record.
2. Does the lesson specify a target loop and step? If no: it is too abstract; revise or discard.
3. Does the lesson duplicate a prior entry? If yes: confirm the prior entry; do not create a new entry.
4. Is the lesson phrased as a process observation, not a performance assessment? If no: revise.
5. Would knowing this lesson have changed the outcome of this cycle? If no: it is an anecdote; record in the Reflection only.

---

### Step 6 — Capture Reusable Engineering Knowledge

**Agent:** `REFLECT-AGENT`, reviewed by `REFLECT-CHECKER`  
**Inputs:** Candidate lesson list, candidate pattern list, evidence index, prior knowledge base  
**Outputs:** Confirmed entries written to `lessons-learned.md`, `engineering-patterns.md`, `decision-log.md`  

**Phase 1 (REFLECT-AGENT):** Write all candidate lessons and patterns that have passed the quality checks from Step 5 into staging records. For each entry, resolve prior-entry references: confirm prior entries, mark obsolete entries, or flag conflicts.

**Phase 2 (REFLECT-CHECKER review):** `REFLECT-CHECKER` independently reviews each staged entry for:
- Evidence accuracy: is the cited evidence actually present in the referenced artefact?
- Scope accuracy: is the declared applicability consistent with the evidence (not over-generalised, not unnecessarily narrow)?
- Non-duplication: does any staged entry duplicate or conflict with a prior entry not already handled in Phase 1?
- Mechanism validity: is the causal chain stated in the `mechanism` field coherent and supported by the evidence?

For each staged entry, `REFLECT-CHECKER` records: `accepted`, `rejected` (with reason), or `needs_revision` (with specific revision instruction). `REFLECT-AGENT` addresses all `rejected` and `needs_revision` entries; `REFLECT-CHECKER` re-reviews once. If still rejected on re-review, the entry is not added to the knowledge base and the disagreement is recorded in the Reflection.

**Phase 3:** Write all accepted entries to the accumulative documents.

**Decision log:** For this cycle, write `decision-log.md` with the significant engineering decisions made and their observed outcomes. A significant decision is one that was not fully determined by the plan — a design choice, a trade-off, an assumption acted upon. Each decision record contains: the decision context, the options considered, the option chosen, the rationale at the time, and the LOOP-006 evidence on whether the decision was sound.

---

### Step 7 — Identify Process Improvements

**Agent:** `REFLECT-AGENT`  
**Inputs:** Unsuccessful pattern list, Category G (process) and Category H (automation) assessments, improvement opportunity from prior runs  
**Outputs:** Improvement proposals written to `improvement-opportunities.md`  

For each process improvement identified in Categories G and H, and for each unsuccessful pattern with a declared `prevention_mechanism`, write an improvement proposal:

| Field | Content |
|-------|---------|
| `improvement_id` | `IMP-NNN` |
| `category` | `process`, `automation`, `planning`, `implementation`, `verification`, `tooling`, `documentation` |
| `title` | Concise action phrase (e.g., "Add interface-stability pre-check to LOOP-004 Step 3 for multi-consumer modules") |
| `target_loop` | The loop whose behaviour this improvement would change |
| `target_step` | The specific step within the target loop |
| `current_behaviour` | What the loop currently does at that step |
| `proposed_behaviour` | What the loop should do differently |
| `expected_benefit` | The measurable improvement: reduced gate triggers, reduced deviations, higher confidence score, fewer rework cycles |
| `evidence` | The specific artefact references that motivated this proposal |
| `priority` | `P1` (prevents Critical/High finding or rework cycle), `P2` (improves confidence or reduces gate triggers), `P3` (reduces duration or improves maintainability) |
| `status` | `proposed` (new this cycle), `confirmed` (proposed in prior cycle and re-observed), `in-progress` (a prior proposal being acted on), `resolved` (a prior proposal that has been implemented) |

**Prior improvement deduplication:** Before adding a new proposal, check prior `improvement-opportunities.md` for matching entries. An improvement observed in two consecutive cycles is escalated from `proposed` to `confirmed` and triggers GATE-2 (a recurring improvement opportunity warrants human attention before a third cycle passes it by).

**GATE-1 trigger evaluation:** If any improvement proposal would:
- Change the behaviour of a gate (adding, removing, or changing a gate trigger condition in any loop)
- Change the Maker/Checker separation requirements
- Introduce a new required section to the LOOP-STANDARD
- Change a repository-wide engineering convention

then GATE-1 fires for that proposal. The proposal is recorded with `status: gate_pending` and is not applied to any SKILL file until GATE-1 is cleared.

**Outputs:** Updated `docs/loops/core/STATUS-007.md`  

Record: task ID, run ID, LOOP-006 outcome consumed, lessons recorded (new, confirmed, superseded), patterns recorded, improvement proposals (new, confirmed), reflection confidence score, gate outcomes, elapsed duration, and final status (`completed`, `stopped`, or `failed`).

### Step 9 — Update SKILL Files

**Agent:** `STATUS-WRITER`  
**Inputs:** Calibration observations from Categories A through J  
**Outputs:** Updated `docs/loops/core/SKILL-001.md` through `SKILL-006.md` (where applicable); updated `docs/loops/core/SKILL-007.md`  

For each loop where a calibration observation was made (a specific pattern in that loop's execution that should change how future runs are configured), update the relevant SKILL file with the observation.

SKILL file updates follow these rules:
- Only update SKILL files for loops where specific, evidence-supported calibration observations were made in this cycle.
- Do not update a SKILL file with a generic statement. Each update must reference the specific `LESSON-NNN` or `IMP-NNN` that motivated it.
- Updates are additions, not replacements. Prior SKILL file content is never deleted.
- Updates are marked with the run ID and date so future reflection runs can trace the source.
- If a SKILL file update would implement a GATE-1-pending improvement proposal, the update is deferred until GATE-1 is cleared.

Update `SKILL-007.md` with observations about the reflection process itself:
- Lesson categories that most frequently produced confirmed lessons in this repository
- Evidence types that most frequently required `REFLECT-CHECKER` rejection (signals REFLECT-AGENT calibration opportunities)
- Average reflection confidence by task type
- Time spent per reflection category (signals where analysis is most complex)

### Step 10 — Publish Reflection Artefacts

**Agent:** `STATUS-WRITER`  
**Inputs:** All reflection outputs, metadata  
**Outputs:** `reflection-summary.md`, `reflection-metadata.md`, final `STATUS-007.md` confirmation  

Write `reflection-summary.md` for this cycle:

- **Cycle overview:** Task ID, task type, LOOP-006 outcome, rework cycle count, total engineering duration (sum of all loop durations from STATUS files).
- **Planned versus actual summary:** Three-line comparison of scope, assumption, and quality dimensions from Step 2.
- **Top three lessons:** The three `LESSON-NNN` entries with the highest combined evidence strength and applicability breadth.
- **Top three improvement opportunities:** The three `IMP-NNN` entries with priority P1, or the highest-priority available.
- **Pattern update summary:** Count of new patterns added (`candidate`), patterns confirmed, patterns marked obsolete.
- **Reflection confidence score:** See scoring method below.

**Reflection confidence score (0–100):** Deductions from 100: −10 per missing required input (FR-1), −8 per lesson rejected by `REFLECT-CHECKER` that could not be revised, −5 per category where evidence was insufficient for any observation, −3 per improvement proposal with `priority: P1` that had no prior entry (first observation — inherently uncertain), −2 per non-evidence-based observation that was filtered out by quality checks. The score is recorded in `reflection-metadata.md`.

Write `reflection-metadata.md` with all provenance fields.

Write `docs/reflection/reflections/REFLECTION-007-{run-id}.md` — the loop's own Reflection on its own execution.

Confirm all Stop Conditions. Write final `status: completed` to `STATUS-007.md`.

## Reflection

At the end of every run — completed, failed, or stopped — the highest-active agent produces a Reflection at `docs/reflection/reflections/REFLECTION-007-{run-id}.md`.

The Reflection must contain all ten sections required by LOOP-STANDARD.md §10, plus the following loop-specific additions:

- **Checker Disagreement Record:** Every lesson or pattern that `REFLECT-CHECKER` rejected or required revision on, with the reason and the resolution
- **Evidence Quality Assessment:** Which categories produced well-supported lessons (strong evidence, clear causation) versus which required inference (weaker support, requiring a `candidate` confidence level)
- **Knowledge Base Health:** The ratio of new entries to confirmed entries to superseded entries in this run — a healthy knowledge base has more confirmed entries than new ones over time
- **Improvement Proposal Disposition:** Which P1 proposals are being raised for the first time (candidate) and which are being escalated (confirmed, having been observed in a prior cycle)

## Human Approval Gates

### GATE-1 — Hard Gate: Significant Process Change, New Engineering Standard, or Repository-Wide Convention Change

| Field | Value |
|-------|-------|
| Gate ID | GATE-1 |
| Gate Type | Hard Gate |
| Position in Workflow | After Step 7 (improvement proposal generation), before any SKILL file is updated with the affected proposal |
| Artefact Under Review | The improvement proposal(s) that triggered the gate, the evidence supporting them, and the prior entries they reference |
| Approver | Principal Engineer or Architecture Owner for process changes; Engineering Lead for standard changes; Repository Owner for convention changes |
| Timeout | None — explicit written approval required |
| Approval Denied — Action | The proposal is marked `status: gate_denied` with the reviewer's rationale; the proposal remains in the knowledge base as a record but is not applied to any SKILL file; the loop continues without the denied proposal |
| Audit Trail | Approval or denial recorded in `STATUS-007.md` under `gate_outcomes.GATE-1`; reviewer identity, role, timestamp, and decision rationale recorded |

**Fires when an improvement proposal would:**
- Add, remove, or change a gate trigger condition in any loop
- Change the Maker/Checker separation requirements for any loop
- Introduce a new required section to `LOOP-STANDARD.md`
- Change a repository-wide engineering convention declared in `docs/loops/shared/coding-standards.md` or `docs/loops/shared/engineering-principles.md`
- Supersede an accepted ADR
- Reduce a Hard Gate to a Soft Gate anywhere in the loop chain

**Reviewer guidance:** The proposal record in `improvement-opportunities.md` presents the evidence, the current behaviour, the proposed behaviour, and the expected benefit. The reviewer must determine: (a) whether the evidence adequately supports the proposed change, and (b) whether the proposed change has unintended consequences for loops or tasks not represented in the evidence. Record the decision and rationale in `STATUS-007.md`.

| Field | Value |
|-------|-------|
| Gate ID | GATE-2 |

### Emergency Stop

Any human principal may terminate a running loop by setting `status: emergency_stopped` in `STATUS-007.md`. On emergency stop: the current step is abandoned; all staged knowledge base entries that have not been written are discarded; the loop produces a partial Reflection; `STATUS-007.md` records the step at which the stop was received. Emergency stop in a reflection loop does not roll back any artefacts — partial knowledge is preserved.

## Failure Recovery

### FR-1 — Missing Implementation or Verification Evidence

**Detection:** A required LOOP-005 or LOOP-006 output file is absent or unreadable.  
**Immediate Action:** Record the missing evidence in `STATUS-007.md`. Narrow the reflection scope: categories that depend on the missing evidence are recorded as `limited_scope` rather than `complete`. The missing evidence is itself the first lesson: a lesson of type `process-gap` in Category G (Engineering Process Improvements), recording that the upstream loop did not produce a required handoff artefact.  
**Recovery:** Reflection proceeds with reduced scope. The confidence score is reduced proportionally. If the missing artefact is LOOP-006's `verification-report.md`, Categories C and D cannot be assessed; all other categories proceed with the available evidence.  
**Reporting:** The evidence gap is recorded in `reflection-metadata.md` under `evidence_gaps`.

### FR-2 — Missing Verification Report (Incomplete LOOP-006)

**Detection:** `STATUS-006.md` records `status: stopped` rather than `status: completed`, and no verification report was produced.  
**Immediate Action:** Record the LOOP-006 incomplete status. Categories C and D (verification and architecture) are `limited_scope`. The fact that LOOP-006 did not complete is the most significant lesson of the cycle and must be the first entry in `lessons-learned.md`.  
**Recovery:** Reflection proceeds with the implementation evidence (LOOP-005 outputs) and the partial verification evidence that was collected before LOOP-006 stopped. The confidence score reflects the incomplete input.  
**Reporting:** Recorded in `reflection-summary.md` as a cycle with `incomplete_upstream_chain`.

### FR-3 — Conflicting Observations

**Detection:** Two categories produce contradictory observations (e.g., Category A identifies a planning gap that Category B identifies as an implementation success; or a lesson from Category C conflicts with a pattern from Category B about the same behaviour).  
**Immediate Action:** Record both observations. Do not resolve the conflict by discarding one.  
**Recovery:** The conflict is presented to `REFLECT-CHECKER` for independent adjudication. If the Checker cannot resolve it with available evidence, the conflicting observations are both recorded with `confidence: candidate` and the conflict is recorded in the Reflection as an `unresolved_conflict`. A GATE-2 notification is sent. The conflict becomes a `knowledge-gap` lesson in Category I.  
**Reporting:** Both observations appear in the knowledge base with a cross-reference annotation `conflicts_with: [LESSON-NNN]`.

### FR-4 — Duplicate Knowledge

**Detection:** A candidate lesson or pattern is identified that is semantically equivalent to an existing entry in the prior knowledge base.  
**Immediate Action:** Do not create a new entry. Confirm the existing entry by adding `confirmed_by: [run-id]` annotation.  
**Recovery:** If the new observation is a refinement (narrower scope, more precise mechanism, or additional evidence) rather than a duplicate, create a new entry that explicitly extends the prior one with `refines: [LESSON-NNN]`. The prior entry remains active.  
**Reporting:** The duplicate resolution is recorded in `reflection-metadata.md` as `confirmed_entries` count.

### FR-5 — Obsolete Guidance

**Detection:** During the prior-knowledge-base review, an existing entry is found to be no longer applicable: the condition it describes no longer exists, the loop it targets has changed, or subsequent cycles have produced contradictory evidence.  
**Immediate Action:** Do not delete the entry. Mark it `status: obsolete` with `obsoleted_by: [run-id]` and `obsolete_reason: [brief explanation]`.  
**Recovery:** If the entry was being used as active guidance in a SKILL file, add a note to the relevant SKILL file that the guidance has been superseded. If the SKILL file update is a significant change, trigger GATE-1.  
**Reporting:** Obsolete entries are counted in `reflection-metadata.md` as `obsoleted_entries`.

### FR-6 — Incomplete Lessons Learned

**Detection:** One or more reflection categories cannot produce any lessons because the available evidence is too sparse to support a conclusion.  
**Immediate Action:** Record the category as `insufficient_evidence` rather than forcing a lesson. The insufficiency itself is a lesson in Category I (Knowledge Gaps): the process did not produce enough evidence in this area for reflection to be informative.  
**Recovery:** The category is not blocked — it is marked with scope `limited`. If the same category is `limited` in three consecutive reflection runs, a GATE-2 notification fires: a persistent evidence gap is an engineering process problem, not a data problem.  
**Reporting:** The evidence insufficiency is recorded in `reflection-metadata.md` under `limited_categories`.

## Risks

### RISK-1 — Recording Opinions Instead of Evidence

- **Description:** Lessons are recorded as general engineering wisdom or personal preference rather than as observations grounded in evidence from this cycle.
- **Likelihood:** Medium
- **Impact:** High — opinion-based lessons corrupt the knowledge base and reduce the reliability of future SKILL file calibrations.
- **Control:** Principle 1 (Evidence-Based Reflection) and Step 5's five-point lesson quality check. `REFLECT-CHECKER` independently reviews every lesson for evidence accuracy. GATE-2 fires if more than three opinion-based lessons are caught in one cycle.
- **Detection:** `refl.checker_rejections` metric elevated; `REFLECT-CHECKER` rejection reason reads "no evidence citation."
- **Response:** Rejected lessons are not recorded. The REFLECT-AGENT calibration issue is itself a lesson in Category G.

### RISK-2 — Duplicate Knowledge

- **Description:** The same lesson is recorded multiple times across cycles, inflating the knowledge base volume without increasing its informational value.
- **Likelihood:** Medium
- **Impact:** Medium — duplicates degrade knowledge base readability and make it harder to identify truly new lessons.
- **Control:** Step 6 Phase 1 explicitly deduplicates against the prior knowledge base before writing. FR-4 procedure applies for detected duplicates. Principle 7 (Non-Duplication) is declarative.
- **Detection:** `refl.lessons_confirmed` is high relative to `refl.lessons_new` in later cycles (this is a sign of a healthy, converging knowledge base) versus a knowledge base where `refl.lessons_new` keeps growing without confirmation (signals deduplication failure).
- **Response:** FR-4 procedure — confirm rather than duplicate.

### RISK-3 — Stale Engineering Guidance

- **Description:** Prior lessons or patterns in the knowledge base become outdated as the repository evolves, but remain active in SKILL files and influence future loop runs incorrectly.
- **Likelihood:** Low over short periods; Medium over many cycles.
- **Impact:** High — stale guidance causes future loops to apply outdated rules.
- **Control:** Each LOOP-007 run reviews prior entries for obsolescence (FR-5 procedure). Entries not confirmed in ten cycles are marked `unconfirmed`. Principle 8 (Long-Term Maintainability) requires explicit obsolescence marking rather than silent rot.
- **Detection:** A lesson from a prior cycle contradicts evidence from the current cycle.
- **Response:** FR-5 procedure — mark `obsolete`; update SKILL files if needed.

### RISK-4 — Over-Documentation

- **Description:** The knowledge base accumulates entries at a rate that makes it impractical to read, leading to future loops ignoring it rather than applying it.
- **Likelihood:** Low initially; Medium over many cycles.
- **Impact:** Medium — an unread knowledge base provides no learning value.
- **Control:** Principle 3 (Reusable Knowledge Capture) filters out anecdotes. Principle 7 (Non-Duplication) prevents volume inflation. Entries are marked `obsolete` or `unconfirmed` rather than growing indefinitely. GATE-2 fires if the reflection confidence score is low, which correlates with low-quality entry generation.
- **Detection:** `refl.lessons_new` grows while `refl.lessons_confirmed` remains near zero across consecutive runs.
- **Response:** A dedicated aggregation reflection run (trigger type 4) reviews the knowledge base for entries to supersede or consolidate.

### RISK-5 — Missing Important Lessons

- **Description:** A significant lesson from the cycle — particularly from a Critical/High finding or a rework event — is not captured because the evidence is present but the analysis was insufficiently deep.
- **Likelihood:** Low
- **Impact:** High — a missed lesson from a rework event means the cycle recurs.
- **Control:** Category G specifically analyses process failures and every GATE-1 trigger. `REFLECT-CHECKER` independently reviews the lesson list for coverage of all Critical/High findings. The first lesson quality check asks: "Would knowing this have changed the outcome?"
- **Detection:** A subsequent cycle encounters the same GATE-1 trigger condition without a prior `LESSON-NNN` preventing it.
- **Response:** The missed lesson is captured in the subsequent cycle as a `confirmed` lesson (observed twice) with elevated priority.

### RISK-6 — Conflicting Standards

- **Description:** Two improvement proposals generated in the same or successive cycles produce conflicting recommendations for the same loop and step.
- **Likelihood:** Low
- **Impact:** High — applying conflicting guidance degrades process quality.
- **Control:** FR-3 procedure handles conflicting observations within a cycle. Step 6's deduplication handles cross-cycle conflicts. GATE-1 applies to all proposals that would change loop standards, providing a human review layer.
- **Detection:** Two `IMP-NNN` entries with the same `target_loop` and `target_step` but contradictory `proposed_behaviour`.
- **Response:** FR-3 procedure — both recorded with `confidence: candidate`; conflict flagged for human resolution.

### RISK-7 — Unverified Recommendations

- **Description:** An improvement proposal is applied to a SKILL file without adequate evidence that the proposed change would actually improve outcomes.
- **Likelihood:** Low
- **Impact:** Medium — an unverified change degrades future loop performance.
- **Control:** All improvement proposals require an `evidence` field citing specific artefact references. GATE-1 applies to proposals that would change process standards. `confidence: candidate` proposals are not applied to SKILL files until confirmed in a second cycle.
- **Detection:** A SKILL file change lacks a `LESSON-NNN` or `IMP-NNN` reference (caught by VER-7).
- **Response:** VER-7 failure prevents the run from being marked completed; the unsupported SKILL file change is rolled back.

### RISK-8 — Tenant Isolation Breach

- **Description:** Not applicable. This loop reads artefacts and writes to `docs/reflection/`. It makes no changes to source code, configuration, or authentication logic.
- **Likelihood:** N/A
- **Impact:** N/A

## Future Improvements

- **Cross-task pattern aggregation:** Extend the aggregation trigger (trigger type 4) with a structured aggregation pass that groups lessons by task category and technology class, identifying cross-task patterns that are invisible at the single-task level — for instance, that all `FEAT` tasks in a specific module consistently produce planning assumption refutations, indicating a LOOP-002 context gap for that module.
- **Knowledge base health scoring:** Produce a per-category health score for the knowledge base (ratio of confirmed to unconfirmed to obsolete entries) that LOOP-002 uses to weight context selection — categories with high health scores produce more reliable lessons and should be preferentially surfaced in future context assemblies.
- **Improvement proposal lifecycle tracking:** Connect `IMP-NNN` proposals to the LOOP-003 task catalog — a confirmed P1 proposal automatically generates a candidate `DOCS` or `RFCT` task in the backlog for the process change, rather than requiring a human to translate the proposal into a task.
- **Lesson applicability refinement:** Track, across cycles, which lessons were surfaced as context by LOOP-002 and which actually influenced the planning or implementation of subsequent tasks — creating a feedback loop from application back to the lesson's confidence score.
- **Temporal obsolescence:** Add a `last_confirmed_run_id` field to every knowledge base entry, enabling a scheduled aggregation run to automatically flag entries that have not been confirmed in the last N cycles as `unconfirmed`, without requiring a human to audit the full knowledge base.
