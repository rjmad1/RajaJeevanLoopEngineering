# Glossary

This glossary defines standard business, software engineering, AI/LLM, and repository-specific terms used in the **RajaJeevanLoopEngineering Framework**.

---

## 1. Business & Product Management Terms

*   **Acceptance Criteria:** The specific, falsifiable conditions that a software product feature must satisfy to be accepted by a product owner or stakeholder.
*   **Epic:** A large body of work that can be broken down into smaller, actionable tasks (User Stories) based on customer needs.
*   **Emergency Stop:** A standing safety protocol that allows any authorized human operator to immediately terminate a running process regardless of its current state.
*   **Hard Gate:** A mandatory checkpoint in an execution workflow that completely halts progress until an authorized human provides written, logged approval.
*   **KPI (Key Performance Indicator):** A quantifiable metric used to evaluate the success and performance of an organization, project, or process.
*   **Maker/Checker Pattern:** A governance principle requiring that any critical or high-risk output generated by one entity (the Maker) be validated by an independent reviewer (the Checker) before it is finalized or merged.
*   **MVP (Minimum Viable Product):** The simplest version of a product that has enough value to be released to early users for feedback.
*   **Product Capability:** A high-level business feature or service category that addresses a user need.
*   **Persona:** A detailed representation of a target user type (e.g., Developer, Product Manager, QA) mapping their goals, pain points, and tasks.
*   **Roadmap:** A strategic planning document mapping product features and capabilities against a timeline.
*   **SLA (Service Level Agreement):** A commitment defining the expected performance level of a system, such as API uptime or response speed.
*   **Soft Gate:** A workflow checkpoint that notifies human operators of progress and sets a timer; if no human registers an objection within the timeout window, the system automatically proceeds.
*   **Stakeholder:** An individual or group affected by the outcome of a project.
*   **User Story:** An informal, natural language description of a software feature written from the perspective of an end-user.
*   **Workflow:** A sequence of tasks that represents a business process or automated execution run.

## 2. Software Engineering Terms

*   **API (Application Programming Interface):** A set of protocols and definitions enabling separate software components to communicate with each other.
*   **Artifact:** A discrete file or record produced during a software build or run (e.g., compiled code, test reports, markdown documents).
*   **Agnostic State Engine:** A software service designed to track and transition execution phases independently of the language or stack running the tasks.
*   **Build Pipeline:** A set of automated processes that compile, test, and package source code into deployable artifacts.
*   **CI/CD (Continuous Integration / Continuous Deployment):** A practice of automating the building, testing, and deployment of code changes.
*   **Circuit Breaker:** A design pattern that monitors execution failures and automatically blocks operations (opening the breaker) if failures reach a threshold, preventing resource exhaustion.
*   **Configuration:** Externalized values (e.g., environment variables, property files) that parameterize software execution without code changes.
*   **Container:** A lightweight, standalone package containing all dependencies needed to run an application, isolating it from the host OS.
*   **Dependency Injection:** A design pattern where an object's dependencies are supplied by an external framework rather than created by the object itself.
*   **Docker:** A containerization platform used to package, run, and isolate services.
*   **Idempotency:** A property of operations where executing a request multiple times yields the same result as executing it once, preventing duplicate side-effects.
*   **Integration Test:** A test verifying that multiple independent modules or services interact correctly.
*   **Middleware:** Software that sits between an operating system or database and the applications running on it, handling requests, security, or routing.
*   **Namespace / Package:** A scoping mechanism organizing source code symbols and preventing naming collisions.
*   **Repository:** A storage location for source files, documentation, and metadata tracked by a version control system like Git.
*   **Serialization:** The process of converting an in-memory object into a byte-stream or text format (e.g., JSON/YAML) for storage or transmission.
*   **Telemetry:** Automated collection and transmission of application traces, metrics, and logs for monitoring and debugging.
*   **Unit Test:** An isolated test verifying the correctness of a single class or function.

## 3. AI / LLM Terms

*   **AI Agent:** An autonomous or semi-autonomous software component that parses instructions, selects tools, and executes code to achieve a target goal.
*   **Chain of Thought (CoT):** A prompting technique encouraging LLMs to break down complex reasoning steps sequentially before generating the final answer, improving correctness.
*   **Context Window:** The maximum amount of text token data an LLM can ingest and process in a single API request.
*   **Embedding:** A vector representation of text capturing its semantic meaning, enabling mathematical comparison and search.
*   **Evaluation (Eval):** Systematic testing of LLM responses against benchmark inputs and expected structures to measure accuracy, safety, and drift.
*   **Function / Tool Calling:** The capability of an LLM to output a structured JSON command mapping to a specific application tool or API rather than plain text.
*   **Hallucination:** A phenomenon where an LLM generates factually incorrect or syntactically invalid output that is not backed by the input context.
*   **Prompt:** The text instructions or queries sent to an LLM to guide its output.
*   **Prompt Engineering:** The practice of structuring, refining, and evaluating prompts to maximize LLM performance and reliability.
*   **Prompt Template:** A parameterized prompt outline containing placeholders (e.g., `{file_content}`) that are populated programmatically before invocation.
*   **Retrieval:** The phase in RAG systems where relevant documents are queried from an external database or index.
*   **RAG (Retrieval-Augmented Generation):** Enhancing LLM outputs by retrieving and injecting relevant documents from an external search index or database into the prompt context.
*   **System Prompt:** Instruction guidelines injected into the LLM context to enforce behavior, formatting, and usage bounds.
*   **Token:** The fundamental unit of text (often a word or partial word) parsed by an LLM during inference.
*   **User Prompt:** The dynamic, task-specific input provided to the LLM by the end user or agent script.
*   **Vector Store:** A specialized database optimized for storing and querying high-dimensional embedding vectors.

## 4. Repository-Specific Terms

*   **Bootstrap CLI:** A script (`bootstrap.py` or `interactive-bootstrap.ps1`) or Java executable (`BootstrapCli`) that clones and initializes Loop documents, Gradle configs, and Dev Containers in a target project.
*   **Loop:** A repeatable unit of AI-assisted engineering work governed by structured pre- and postconditions.
*   **Loop Run:** A single execution instance of a Loop document, tracking state from initial discovery to reflection.
*   **LoopSpecCompiler:** A tool that compiles JSON/YAML loop declarations into conformant, standardized Markdown files.
*   **Reflection Log:** A retrospective document (`REFLECTION-NNN.md`) produced at the end of a Loop Run summarizing actions, outcomes, metrics, and failures.
